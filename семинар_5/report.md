# Отчёт 

## 1. Новый инструмент

Сигнатура `compare_periods` (в `tools.py`, описан в `schemas.py`):

```python
def compare_periods(
    metric: str,        # "key_rate" | "fx_USD" | "fx_EUR" | "fx_CNY" | "cpi" | "unemployment"
    period_a: str,      # "YYYY-MM" или "YYYY-MM-DD"
    period_b: str,
) -> dict:
    """
    {"metric": ..., "a": {"date","value"}, "b": {"date","value"},
     "delta": b.value - a.value, "ratio": b.value / a.value, "source": ...}
    """
```

Вызывает `get_fx_rate / get_key_rate / get_inflation /
get_unemployment. Период `YYYY-MM` разворачивается в
последний день месяца; ошибка возвращается как `{"metric": ..., "error": ...}`.

Примеры запросов, где инструмент вызывается:

| Запрос | Вызов | Результат |
|---|---|---|
| «Во сколько раз вырос курс USD с января 2022 по апрель 2026?» | `compare_periods("fx_USD","2022-01","2026-04")` | a=77.82, b=74.88, **ratio≈0.96** (снизился) |
| «На сколько п.п. изменилась ключевая ставка с дек. 2021 по март 2026?» | `compare_periods("key_rate","2021-12","2026-03")` | a=8.5, b=16.0, **delta=+7.5 п.п.** |
| «Как изменилась инфляция г/г между мартом 2024 и мартом 2026?» | `compare_periods("cpi","2024-03","2026-03")` | a=7.72, b=6.80, **delta=−0.92** |

## 3. Диагностика: три разных типа ошибок

### Тип 1. Битый JSON в аргументах tool-call
Трасса:
```json
{"step": 1, "call": "get_fx_rate", "args": {},
 "obs": {"error": "битый json аргументов: Expecting ',' delimiter: line 1 column 44 (char 43)"}}
```
**Причина.** Модель прислала аргументы tool-call как невалидный JSON (оборванная
строка без `}`). `json.loads` падает, исполнитель ловит `JSONDecodeError` —
инструмент даже не вызывается. Корень — в декодировании аргументов модели.
**Что исправить.** Включить json-mode / constrained-decoding для аргументов;
при битом JSON слать модели подсказку «перешли валидный JSON по схеме» и делать
один ретрай.

### Тип 2. Галлюцинация инструмента
Трасса:
```json
{"step": 2, "call": "get_gdp", "args": {"year": 2024, "quarter": 3},
 "obs": {"error": "неизвестный инструмент: get_gdp"}}
```
**Причина.** Модель выдумала инструмент `get_gdp`, которого нет в схемах (ВВП в
наборе не предусмотрен); `TOOLS_IMPL.get(name)` вернул `None`. Корень — пробел в
покрытии: метрики нет, а модель «дорисовала» функцию вместо отказа.
**Что исправить.** В системный промпт правило «звать только инструменты из
списка; нет данных — честно сказать»; опционально — валидировать имя вызова
против схем до исполнения.

### Тип 3. Инструмент упал на плохих аргументах
Трасса:
```json
{"step": 3, "call": "get_fx_rate", "args": {"ccy": "USD"},
 "obs": {"error": "плохие аргументы для get_fx_rate: ... unexpected keyword argument 'ccy'. Expected: {'currency': 'str', 'on_date': ...}"}}
```
**Причина.** Имя инструмента верное, но модель передала параметр `ccy` вместо
`currency`. Python поднимает `TypeError`, исполнитель кладёт его в `obs.error`
вместе с ожидаемой сигнатурой.
**Что исправить.** Строже описать параметры в схеме (точные имена, `required`);
в обработчике подсказывать ожидаемые поля (уже частично есть через `Expected`)
+ один ретрай на исправление.


## 4. Вывод: самый хрупкий элемент системы

Считаю самым хрупким элементом - **границы данных**: агент склонен запрашивать недоступные периоды и спасается только встроенным ретраем, тратя лишние вызовы. Наибольший прирост надёжности дали бы constrained-decoding аргументов с ретраем по схеме и явные правила в промпте — звать только существующие инструменты и при нехватке данных брать ближайший доступный период, а не выдумывать число.

# Финальный проект — Матчинг резюме ↔ вакансия с проверяемым обоснованием

**Трек B (прикладной).** Система оценивает соответствие кандидата вакансии и
выдаёт структурированное обоснование с дословными цитатами из резюме, проверяя
их на галлюцинации. Бэкенд — DeepSeek (`deepseek-chat`). Всё руками, без
фреймворков; интерфейс — CLI.

> Числа в разделах 3–4 подставляются скриптом `fill_report.py` из
> `output/eval_results.json` после прогона `eval.py`.

---

## 1. Задача и данные

**Задача.** Дана пара (резюме кандидата, описание вакансии). Нужно: (а) выдать
метку соответствия `No Fit / Potential Fit / Good Fit`; (б) обосновать её —
перечислить совпавшие и недостающие навыки и подтвердить каждое совпадение
**дословной цитатой из резюме**; (в) не допустить выдуманных обоснований.
Прикладная ценность — предварительный скрининг откликов с проверяемым следом
решения (почему кандидат подходит), а не «чёрный ящик».

**Данные.** Реальный публичный датасет Hugging Face
`cnamuangtoun/resume-job-description-fit`: колонки `resume_text`,
`job_description_text`, `label` ∈ {No Fit, Potential Fit, Good Fit}. Объём:
train 6241, test 1759; резюме длинные (медиана ~5000 символов), вакансии ~2400.
Для eval берём **стратифицированную** выборку из `test.csv` (по равному числу пар
на каждый из трёх классов; по умолчанию 6×3 = 18 ≥ 15). Метки датасета
используются как gold для измерения правильности. Данные сторонние, берутся как
есть; персональные поля не добавляем и в обоснованиях не используем.

---

## 2. Система

Техники курса (≥4, реально работают):
- **RAG** — `retrieval.py`: Okapi BM25 (написан руками) по фрагментам резюме;
  под каждое требование вакансии достаём релевантные куски, а не весь текст.
- **Агент с инструментами** — `agent.py`: ReAct-цикл с инструментами
  `search_resume` и `check_skill_present` (`tools.py`).
- **Мультиагент** — матчер + **скептик** (`critic.py`), который перепроверяет
  оценку на завышение и может её понизить.
- **Структурированный вывод** — `schema.py`: `response_model` + `max_retries`,
  бизнес-инварианты через `field_validator`/`model_validator` (см. ниже).
- **LLM-as-judge** — `judge.py`: независимая оценка обоснованности (1..5),
  ортогональная gold-метке.
- **Проверка галлюцинаций** — `hallucination.py`: детектор ghost-цитат/навыков.

**Бизнес-инвариант (`field_validator`/`model_validator`).** `confidence ∈ [0,1]`;
`min_years_experience ∈ [0,60]`; оценки судьи `1..5`; ключевой
`model_validator`: вердикт **«Good Fit» недопустим без совпавших навыков и без
доказательств-цитат** — модель не может поставить высшую метку без обоснования.

**Путь данных (5–7 строк).**
1. Из вакансии извлекаются структурированные требования (`JobRequirements`).
2. Резюме чанкуется и индексируется BM25.
3. Матчер-агент по каждому требованию зовёт `search_resume`/`check_skill_present`,
   собирая факты и цитаты.
4. Из транскрипта формируется структурный `FitAssessment` (метка + доказательства).
5. Детектор галлюцинаций отбрасывает ghost-цитаты и **понижает** вердикт, если
   доказательств не осталось.
6. Скептик перепроверяет на завышение; итог — минимум из двух мнений.

## 3. Оценка

Eval на **18 (успешно оценено 18, ошибок API 0)** парах (стратиф. по меткам). Правильность — против gold-меток
датасета; путь — шаги/инструменты/токены; плюс LLM-as-judge и галлюцинации.

| Метрика | Значение |
|---|---|
| Exact accuracy | **0.389** |
| Adjacent accuracy (ordinal) | **0.833** |
| Macro-F1 | **0.381** |
| Судья: groundedness (1..5) | 3.56 |
| Судья: relevance (1..5) | 3.72 |
| Среднее шагов агента | 16.61 |
| Среднее вызовов инструментов | 23.17 |
| Вызовов LLM / токенов | 135 / 209153 |
| Ориентировочная стоимость | $0.0793 |

**Матрица ошибок** (строки — gold, столбцы — pred):

| gold \ pred | No Fit | Potential Fit | Good Fit |
|---|---|---|---|
| No Fit | 4 | 2 | 0 |
| Potential Fit | 5 | 1 | 0 |

Комментарий: exact accuracy уместно дополнять adjacent — классы порядковые
(No < Potential < Good), и ошибка на соседний класс мягче, чем
No↔Good. Судья оценивает обоснованность отдельно от попадания в метку.

## 4. Где пайплайн ошибался

**Проверка галлюцинаций (с цифрой).** Доля ответов хотя бы с одной ghost-цитатой:
**0.278**; всего ghost-цитат: **4** из 23
(rate 0.174); ghost-навыков: 3. Эти случаи детектор
поймал и понизил по ним вердикт.

Конкретные провалы (вход → искажение):

- **GoodFit_1**: gold `Good Fit` → предсказано `Potential Fit`. matched=['C++', 'algorithms', 'distributed systems'], missing=['Python', 'data structures', 'cloud services', 'ML infrastructure']. Вероятная причина — лексический промах ретрива/разногласие разметки.
- **GoodFit_4**: gold `Good Fit` → предсказано `No Fit`. matched=['Java', 'SQL'], missing=['Microservices', 'RESTful API', 'Relational Databases', 'Distributed Systems', 'Agile', 'TDD']. Вероятная причина — лексический промах ретрива/разногласие разметки.
- **NoFit_1**: модель привела цитату-призрак (нет в резюме): «skill:Office Suite…» — детектор отбросил её и понизил вердикт.

## 5. Выводы

- **Что работает.** Связка «RAG-доказательства + ghost-детектор + скептик»
  заметно снижает риск завышенных вердиктов: высшая метка требует дословного
  подтверждения, а выдуманные цитаты отсекаются и понижают оценку. Структурный
  вывод с `model_validator` гарантирует, что «Good Fit» всегда обоснован.
- **Границы.** Метки датасета субъективны (что считать Potential vs Good —
  спорно), поэтому часть «ошибок» — это разногласие с разметкой, а не брак
  системы; отсюда смысл adjacent-accuracy. BM25 — лексический, он промахивается
  на синонимах/перефразировках навыков (например «ML» vs «machine learning»),
  что бьёт по recall совпадений. Скептик иногда излишне консервативен и роняет
  пограничные «Good Fit» в «Potential».
- **Что в прод, что переделать.** В прод оставил бы ghost-детектор и требование
  цитат — это даёт аудируемость. Переделал бы ретрив: добавить эмбеддинги/
  синонимические словари навыков вместо чистого BM25; вынести порог скептика в
  конфиг; калибровать confidence. Судью на DeepSeek — оставить только для
  оффлайн-оценки, не в онлайн-путь (стоимость/латентность).

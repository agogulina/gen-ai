# Финальный проект — Матчинг резюме - вакансия с проверяемым обоснованием

**Трек B (прикладной).** Система оценивает соответствие кандидата вакансии и
выдаёт структурированное обоснование с дословными цитатами из резюме, проверяя
их на галлюцинации.  Артефакты в `output/`.

---

## 1. Задача и данные

**Задача.** Дана пара (резюме, описание вакансии). Нужно: (а) выдать метку
соответствия (`Fit` / `No Fit`; в трёхклассовом режиме — `No Fit` /
`Potential Fit` / `Good Fit`); (б) обосновать её — перечислить совпавшие и
недостающие навыки и подтвердить каждое совпадение **дословной цитатой из
резюме**; (в) не допустить выдуманных обоснований. Прикладная ценность —
предварительный скрининг откликов.

**Данные — два набора**

1. **Синтетический датасет (основной), 18 пар, бинарная разметка Fit/No Fit.**
   Сгенерирован: для `Fit` резюме содержит навыки,
   покрывающие требования вакансии (например, QA-инженер с Python/Selenium/
   Jenkins на вакансию QA-автоматизатора); для `No Fit` берётся резюме из
   другой профессии. Баланс 9/9.  Лежит в `input/eval_set.jsonl`.

2. **Публичный датасет (контрольный), 3 класса.** `cnamuangtoun/resume-job-
   description-fit` (HF), реальные резюме и вакансии, метки `No Fit / Potential /
   Good`. Использован как стресс-тест на «диких» данных с
   субъективной разметкой. Результаты приложены отдельно в `результаты_прошлые/`.
---

## 2. Система

Техники курса:
- **RAG** (`retrieval.py`): Okapi BM25; под каждое требование
  достаём релевантные фрагменты длинного резюме, а не весь текст.
- **Агент с инструментами** (`agent.py`, `tools.py`): ReAct-цикл с инструментами
  `search_resume` и `check_skill_present`.
- **Мультиагент** (`critic.py`): матчер + скептик, перепроверяющий оценку на
  завышение и способный понизить вердикт.
- **Структурированный вывод** (`schema.py`): `response_model` + `max_retries`,
  бизнес-инварианты через валидаторы.
- **LLM-as-judge** (`judge.py`): независимая оценка обоснованности (1..5).
- **Проверка галлюцинаций** (`hallucination.py`): детектор ghost-цитат/навыков.

**Бизнес-инвариант (`field_validator`/`model_validator`).** `confidence ∈ [0,1]`;
`min_years_experience ∈ [0,60]`; оценки судьи `1..5`; ключевой `model_validator`:
вердикт «Good Fit» недопустим без совпавших навыков и без доказательств-цитат.

**Путь данных.**
1. Из вакансии извлекаются структурированные требования (`JobRequirements`).
2. Резюме чанкуется и индексируется BM25; считается **детерминированное покрытие**
   обязательных навыков (доля найденных в резюме).
3. Матчер-агент инструментами собирает факты и цитаты по требованиям.
4. Формируется структурный `FitAssessment` (метка + доказательства).
5. Детектор галлюцинаций отбрасывает ghost-цитаты и при нехватке доказательств
   понижает вердикт.
6. Скептик перепроверяет на завышение; итог + обоснование пишутся в `output/`.

## 3. Оценка

Eval считает **правильность** (по gold-меткам) и **путь** (шаги/инструменты/
токены/стоимость), плюс LLM-as-judge и метрики галлюцинаций.

### Основной прогон — синтетический датасет (бинарный, 18 пар)

| Метрика | Значение |
|---|---|
| Exact accuracy | **1.000** |
| Precision / Recall / F1 (класс Fit) | 1.0 / 1.0 / 1.0 |
| Macro-F1 | 1.000 |
| Судья: groundedness / relevance | **4.67 / 4.83** |
| Среднее шагов агента / вызовов инструментов | 10.2 / 14.8 |
| Вызовов LLM / токенов / стоимость | 125 / 151 435 / **$0.057** |
| Ошибок API | 0 |

Матрица ошибок (синтетика):

| gold \ pred | No Fit | Fit |
|---|---|---|
| No Fit | 9 | 0 |
| Fit | 0 | 9 |

1.0 подтверждает, что **конвейер собран правильно и сквозно работает** (извлечение → RAG → агент → структурный вывод →
проверка → судья), но это **верхняя граница на чистой разделимой задаче**, а не
доказательство решения «дикого» матчинга. Поэтому ниже — контрольный прогон на реальных данных.

### Контрольный прогон — публичный датасет (3 класса, `результаты_прошлые/`)

| Метрика | Значение |
|---|---|
| Exact accuracy | 0.389 |
| Adjacent accuracy | **0.833** |
| Macro-F1 | 0.381 |
| Судья: groundedness / relevance | 3.56 / 3.72 |

Здесь exact ≈ уровню случайного угадывания для 3 классов, но adjacent 0.833 —
почти все ошибки на соседний класс (No↔Potential, Potential↔Good), грубых No↔Good мало. Это честный результат на субъективной разметке.

## 4. Где пайплайн ошибался

**Синтетика: галлюцинации цитат при идеальной классификации (главный честный
вывод).** Хотя все 18 меток предсказаны верно, слой «доказательств» далёк от
идеала: детектор поймал **17 ghost-цитат из 52 (rate 0.327)**, ghost хотя бы в
одном ответе — у **6 из 18** пар (в основном в `Fit`-парах, где модель пыталась
процитировать резюме и иногда пересказывала фразу неточно или дописывала
несуществующую). Эти цитаты не прошли дословную сверку с резюме и были отброшены.
Вывод: **правильная метка ≠ честное обоснование** — без антигаллюцинационного
контура система выдавала бы выдуманные «подтверждения».

**Публичные данные: 3 конкретных провала (вход → искажение).**
- **GoodFit_1** (gold Good → pred Potential): подтверждены `C++`, `algorithms`,
  `distributed systems`, но `Python`/`cloud services` записаны иначе и попали в
  missing → метка занижена. Навыковый сигнал не увидел синонимичный опыт.
- **GoodFit_4** (gold Good → pred No Fit): подтверждены `Java`/`SQL`, но требуемые
  `Microservices`/`RESTful API` отсутствуют дословно → система решила «почти всё
  missing». Кандидат профильный, но формулировки разошлись.

Эти случаи показывают системную причину низкой exact на публичных данных:
холистическая метка «fit» **не сводится** к пересечению навыковых токенов.

## 5. Выводы

- **Ответ по задаче.** Система-скрининг с проверяемым обоснованием собирается
  руками из техник курса и **сквозно работает**: на контролируемом наборе, где
  метка определяется навыками, классификация идеальна (1.0) при высокой
  обоснованности (judge 4.67). Это подтверждает корректность инженерии.
- **Главная граница (контраст двух наборов).** Та же система на реальном датасете
  даёт exact 0.389 — потому что там метка холистическая (решение человека о
  «пригодности» в целом), а наш ведущий сигнал — пересечение навыков. Узкое место,
  таким образом, **не в коде, а в соответствии признака целевой метке**: где
  «fit» = функция навыков (синтетика) — точность предельная; где «fit» —
  субъективная оценка (публичные данные) — потолок ~0.4 exact / 0.83 adjacent.
  Поэтому для порядковых меток честная headline-метрика — adjacent.
- **Проверка галлюцинаций — ключевая ценность.** Даже при идеальной классификации
  треть цитат-обоснований оказались выдуманными и были пойманы (0.327 на синтетике,
  0.174 на публичных). Без этого контура «обоснование» было бы недостоверным.
- **Что оставить в прод.** Антигаллюцинационный слой (дословные цитаты + ghost-
  детектор + понижение вердикта) и связку «детерминированное покрытие навыков +
  LLM-обоснование»: число решает, LLM объясняет, проверка ловит выдумки.
- **Что переделать.** Лексическое покрытие → семантическое (эмбеддинги/словарь
  синонимов), чтобы не терять навыки «другими словами» (GoodFit_1/4) и не завышать
  на общих навыках; добавить в решение опыт/уровень, а не только навыки;
  калибровать пороги на отдельной выборке; судью использовать только офлайн
  (стоимость/латентность). На реальных «fit»-данных без обучаемой модели потолок
  останется умеренным — это свойство задачи, а не конвейера.
